# Lazy Eye Inspection: Capturing the State of Happy Eyeballs Implementations メモ
- https://arxiv.org/pdf/2412.00263
- https://www.happy-eyeballs.net

## Abstract
- HEのすべての機能をサポートしているのはSafariのみ
  - IPv4 Connection Attempt Delay、Resolution Delay、アドレスの交互配置
- ChromeとFirefox においては、IPv6の設定が完全に機能している場合であっても
  DNSにおいてAレコードのルックアップに問題がある場合、ネットワーク接続が遅延したり中断されたりする

## 1 Introduction
- HEの目的はIPv6 の利用を促進すること
  - IPv6を優先しつつ、必要に応じてIPv4にフォールバックする
- この論文による成果:
  - (i) 7つのOS上の9種類のブラウザにおけるHE実装の評価
    - 多くの実装はHEv1 (RFC 6555) に限定されている
  - (ii) DNSリゾルバにおけるIPの選択とフォールバック時の挙動の測定
    - DNSリゾルバはデュアルスタック構成において一般にHE型のアプローチに依存していない
    - その挙動は非常に多様であり、厳密にIPv6を優先する運用者はほとんど存在しない
  - (iii) 複数のクライアント、異なるバージョンにわたってHE実装をテストするローカルテストフレームワーク
  - (iv) HEが考慮するすべてのプロトコルについてネットワーク遅延をエミュレートするWebベースのテストツール

## 2 Happy Eyeballs
- HEv1 (RFC 6555) は宛先サーバへの接続試行をターゲットとしている
  - ホストのアドレス選択ポリシーに基づきどちらかのアドレスファミリを優先しつつ、
    IPv6 / IPv4アドレス間で接続確立を競争させる
  - DNSが各アドレスファミリについて複数のレコードを返す場合、ホストはアドレス選択ポリシーに従ってそれらを並べ替える
  - 最初のIPv6接続とその後のIPv4接続の両方が失敗した場合の挙動は規定されておらず、アプリケーション開発者に委ねられる
  - 150-250ms以内にIPv6接続が完了しなかった場合、並行してIPv4接続を開始する (CAD: Connection Attempt Delay)
  - いずれか1つの接続試行が成功した場合、クライアントは他の接続試行を破棄し、その結果を10分程度キャッシュすべき
- HEv2 (RFC 8305) は接続試行に加えて名前解決もターゲットとする
  - IPv6の名前解決を優先しつつ、IPv6に問題がある場合にユーザから見える遅延を最小限に抑える
    - クライアントはまずAAAAクエリを発行し、その直後にAクエリを発行する
    - クライアントがAAAAレコードを含む応答を先に受け取った場合、IPv6を使って対象ホストへの接続試行をただちに開始する
      - A応答が先に到着した場合、アプリケーションはAレコードの受信時点から50ms待機する (RD: Resolution Delay)
        - RDの間にクライアントがAAAA応答を受け取った場合、IPv6による接続試行を開始
        - AAAA応答がない場合、IPv4による接続試行を開始
  - アドレスの並び替えにはIPアドレスファミリに加え、過去のTCPラウンドトリップ時間など
    以前に使用したアドレスに関する知識を利用できる
  - リスト内のアドレスは、IPアドレスファミリが交互に現れるように配置される (IPv6優先)
  - 複数のIPv6アドレスが続いても良い (First Address Family Count分: 1~2)
  - 接続試行の間にはCADを使用する
    - CADとして固定値250 msを推奨 (10ms~2s)
- HEv3 (I-D) はプロトコルの検出をサポートするため、SVCB / HTTPS DNSリソースレコードを処理する
  - HEv3のアドレス選択では利用可能なTLS Encrypted ClientHello (ECH) を持つIPアドレス -> QUIC ->  TCPを優先する

## 3 Related Work
- 現実において、RFCの推奨事項がほとんど使われておらず、HEは部分的にしか実装されていない
- IPv6のみのインターネットへ向かう傾向はありつつも、ネットワークの重要な構成要素には完全なIPv6サポートが欠けている
- インターネットにおける体験品質にとって、依然としてHEv1/HEv2は重要な仕組み
- IPv6の展開に時間がかかっている一方でQUICや最初のECH情報を含むHTTPS RRは早急かつ広範に展開しており、
  これらの可能性を考慮したHEv3の早期標準化は妥当
- HE全体のアルゴリズムがより複雑になる場合、クライアントの実装はRFCの推奨事項からさらに乖離する可能性がある

## 4 Measurement Methodology
### 4.1 Measurement Targets
- Connection Attempt Delay (CAD)
  - CADを取得するため、あるドメイン名がIPv4 / IPv6アドレスに解決されるように設定し、
    サーバ側でtc-netemを用いてIPv6 パケットを遅延させることによりIPv6接続が遅い状況をシミュレートする
- Resolution Delay (RD)
  - RDを取得するため、エンコードされたテストパラメータに従って応答を遅延させる独自実装の権威ネームサーバを使用する
    - パラメータは遅延時間、遅延させるリソースレコードの種類、キャッシュの影響を防ぐためのnonceを設定できる
- Address Selection
  - 宛先アドレス選択を評価するため、各ファミリにまったく応答しないアドレスを含む複数のアドレスを返す

### 4.2 Recursive Resolver
- DNSリゾルバは新しい権威ネームサーバへの各委任において、そのゾーンのネームサーバに接続する際に
  IPv6をIPv4より優先するか、またどのように優先するかを判断する
- 本論文ではリゾルバの一般的なIP優先挙動と、HEのアプローチを使用しているかどうかを確認する
  - ゾーンを提供する権威ネームサーバに対してトラフィックシェーピングを適用する
    - RD測定の場合にはネームサーバレコードに対して、CAD測定の場合には対応するIPアドレスに対して適用される
  - 単一ゾーン内で異なるドメイン名を用意するのではなく、測定する遅延ごとに完全に別々のゾーンを作成する

### 4.3 Measurement Setups
- テストケースは二つの異なる環境で行う
  - (i) Local Testbed Framework
    - クライアントとサーバの2台のホストを直接接続したローカルテストベッド
  - (ii) Web-based Testing Tool
    -  7つのOS上の9種類のブラウザ

## 5 Evaluation

### 5.1 Connection Attempt Delay
- Mozilla Firefox、Google Chrome、Microsoft Edge、Chromium (Ubuntu LTS 24.04.1)
- MacOS上でのSafari 17.5
- curl
- wget

#### ローカル測定
- すべてのクライアントアプリケーションはIPv4/IPv6 の両方が提供された場合にIPv6を優先する
- wget 1.21.3はいかなるフォールバック機構も実装していない (HEを実装していない)
  - wgetはタイムアウトより大きな遅延がある場合、提供されたIPv4 アドレスを使用せずに失敗する
- 観測CADの中央値と標準偏差は、IPv6によって接続が確立された最大遅延値から1ms以内に収まる
- すべてのChromiumベースのブラウザは300msのCADを使用する
- curlは最も小さい200msのCADを使用する
- FirefoxはRFCの推奨である250 msに従っているが、IPv6接続確立を250 ms より長く待つ外れ値がいくつか存在する

#### Web測定
- Safariはローカル測定では一貫して2sであった一方、Webでは50msから最大5sまでの範囲に及び、CAD が動的に適用されている
- Safari以外の他のWebベース測定では通常CAD値まではIPv6が使われるという明確な傾向が見られる
  - Safariは、より小さい遅延ではIPv4を使用し、より大きい遅延では再びIPv6を使用する
  - FirefoxとChrome にも、このような不整合を示すテスト実行が少数あり、10回のテスト反復のうち最大2回しか現れた
  - Safariでは、6~10回の範囲で現れた
    - Safariにおいて、ネットワーク環境、アクセスネットワーク、並行するネットワーク活動の有無、
      アプリケーションウィンドウのフォーカス状態、電源供給の有無のいずれも結果の再現性に目立った影響を与えなかった
    - Mobile Safari の測定では変動するCAD値と不整合が報告された
      - iOSを搭載した携帯電話上のIPv6接続では、CADが1秒を超えることはなかった
        - モバイルデバイスではより小さい値が優先される傾向があると推測される
      - Appleは、プライバシー保護型プロキシサービスであるiCloud Private Relay (iCPR) を展開している
        - iCPRの一部として、SafariはMASQUEプロキシを使用し、egress operatorを経由してトラフィックを中継できる
        - iCPR経由の接続は、リレーネットワークを介してIPトンネルを作成するのではなく、
          接続したいサーバ名だけをegress nodeに通知するため、Safariと比較してまったく異なるHE挙動を示す
          - egress node = DNS解決とTCP / UDPのトランスポートプロトコルヘッダまでのスタックを処理するコンポーネント
        - iCPR経由の測定は、egress operatorがHEをどのように実装しているかを示す
        - AkamaiとCloudflareのegress nodeは、それぞれ150 / 200msのCADを使用
        - ユーザはegress node providerを能動的に選択することはできず、Fastlyはproviderとして現れなかった

### 5.2 Resolution Delay
- RDのテストケースではクライアントがRDをどのように実装しているかを観測するために、AAAAレコードを遅延させる
- 実際にRFCの推奨地である50msのRDを実装しているのはSafariのみ
- Chromiumベースのブラウザ、Firefox、curl、wgetはデフォルト状態ではリゾルバのタイムアウトに依存しており、
  名前解決に関する独自のタイムアウトを適用していない
  - AAAAクエリがリゾルバのタイムアウトに達した後にのみ、IPv4接続を開始する
- Aクエリ応答が遅延した場合のクライアントの反応:
  - 通常HEの文脈では、IPv4は安定したフォールバック先と見なされる一方、IPv6は壊れる可能性のある新しい方式と見なされる
  - 実際にはIPv4も必ず信頼できるわけではなく、Aレコードに対するクエリもIPv6経由で送信される
  - すべてのクライアントはIPv6を優先し続けたものの、Safariを除くすべてのクライアントは常にA応答の到着を待つ
    - AAAAレコードは遅延していないにもかかわらず、A応答が到着して初めて対象ホストへのIPv6接続を開始する
    - この挙動により、一部のリゾルバ構成では高い遅延が発生した場合にChromeとFirefoxの接続が完全に失敗する
      - 2024年4月以降、ChromiumはRDを追加し、この問題のある挙動を解消する機能フラグ (HEv3) を提供している
- iCPR経由のSafari:
  - Cloudflareのegress nodeは、1.75sの遅延まではIPv6を使用する
  - Akamaiのegress nodeは、400msのタイムアウトを使用する
  - 両オペレータはAレコードクエリとAAAAレコードクエリに同じタイムアウトを使用している
  - これらの特性はiCPRを使用しないSafariの挙動とは大きく異なる
  - 名前解決経路上に問題が存在する場合、iCPRはユーザー体験に悪影響を及ぼす可能性がある
- アドレス選択:
  - 10個のAレコードと10個のAAAAレコードを持つドメイン名を用いて測定
  - すべてのレコードは、応答しないアドレスを指すようにする
  - 目に見えるアドレス選択挙動を示したのはSafariのみ
  - その他のクライアントは、IPv4への単純なフォールバックだけを行い、その後停止し、他のレコードのアドレスは無視した

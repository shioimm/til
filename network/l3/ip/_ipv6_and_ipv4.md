# IPv6 / IPv4 共存技術
## デュアルスタック
- ホストがIPv4とIPv6を両方持ち、接続先に応じて使い分ける
- 実装がシンプルで信頼性が高い、IPv4/IPv6どちらとも直接通信できる、
- IPv4アドレスを消費し続ける、管理するプロトコルスタックが2つになる

## トランスレータ
- IPv6-onlyクライアントからIPv4-onlyサーバへアクセスする際、DNSや中間機器やホスト自身がアドレスとパケットを変換する
- エンドツーエンドの透過性が失われる

### DNS64
- IPv4アドレスをIPv6アドレスへ合成する仕組み
- AAAAレコードを受け取ったDNS64リゾルバ (フルリゾルバ) が、Aレコードを元にAAAAレコードを合成して返す
- フルリゾルバへの問い合わせが発生しない場合 (リテラルを扱う場合) は利用できない (それはそう)
- NAT64対応ゲートウェイと組み合わせて利用される

#### 動作フロー
1. IPv6-onlyホストからDNS64対応フルリゾルバにAAAAクエリを問い合わせ
2. フルリゾルバから権威サーバへAAAAを問い合わせ
3. 権威サーバにAAAAレコードが存在する場合、フルリゾルバはAAAAレコードをIPv6-onlyホストへ返答
4. 権威サーバにAレコードしかない場合、フルリゾルバはAレコードを問い合わせ直す
5. フルリゾルバは取得したIPv4アドレスをIPv6 prefixに埋め込んだAAAAレコードを合成する
    - Well-Known Prefix 64:ff9b::/96 またはネットワーク固有プレフィックス
6. フルリゾルバは合成したAAAA をIPv6-onlyホストへ返す
7. IPv6-onlyホストは取得したIPv6アドレスへ接続
8. 経路上のNAT64ゲートウェイがIPv6アドレスをIPv4アドレスへ変換し、IPv4-only サーバへ到達

### NAT64
- IPv6パケットをIPv4パケットとして扱う仕組み
- IPv6パケットを受け取ったネットワーク上のNAT64ゲートウェイがパケットのヘッダを書き換えることで
  IPv6パケットをIPv4パケットへ変換し、IPv4-onlyサーバへ送信する
- 前提としてアドレスの合成が必要 (DNS64もしくは464XLAT / CLATと組み合わせて利用される)
- NAT64装置がSPOF・ボトルネックになりうる

#### 動作フロー
1. DNS64または464XLAT / CLATがサーバのIPv4をプレフィックスに埋め込んだ合成AAAAを作成
2. IPv6-onlyホストがIPv6パケットを送出
3. NAT64ゲートウェイがNAT64 prefixを検出し、IPv4アドレスを抽出し、IPv4パケットを作成してIPv4-onlyサーバへ送信
4. IPv4-onlyサーバがNAT64ゲートウェイへIPv4パケットを応答
5. NAT64ゲートウェイがセッションテーブルを逆引きしてIPv6パケットを作成し、IPv6-onlyホストへ送信

#### PREF64
- NAT64がIPv4アドレスを埋め込むために使用する特別なIPv6プレフィックス (Pref64::/n Discovery [RFC7050])
- well-knownプレフィックスとして64:ff9b::/96がIANAにより定められている
- ISPが独自のプレフィックスを使う場合、ルータ広告オプションでクライアントに通知する ([RFC 8781])

### 464XLAT / CLAT
- IPv4アドレスをIPv6アドレスへ合成する仕組み
- IPv6-onlyホスト自身が宛先IPv4をPREF64に埋め込むことでIPv6アドレスへの合成を行う
- 直接リテラルを扱う場合かIPv4のソケットAPIを直接使うレガシーアプリケーションで利用するのが望ましく
  - ステートレスに合成を行うため、名前解決を行う機会があるのであればDNS64を利用する
- PREF64を得るための手段としてRAのPREF64オプションが利用できる場合、DNS64無しでアドレスを合成できる

#### PREF64オプション
- IPv6のルータ広告 (Router Advertisement) において、ルータがホストへ該当リンクのPREF64を直接知らせるオプション
- 従来PREF64はipv4only.arpaへのAAAAクエリによって取得していたが、これがDNS64に依存する仕組みであるため
  利用できないケースがあることを踏まえた改善策

#### 動作フロー
1. アプリケーションがIPv4パケットを送信
2. CLATがPREF64を使ってアドレス合成し、IPv4パケットをIPv6パケットに変換する
3. CLATがIPv6網を経由してパケットを転送
4. NAT64 (PLAT) が設定されたprefixが宛先となっているパケットのIPv6ヘッダをIPv4ヘッダに置き換えてサーバへ転送する
5. サーバから応答があった場合、NAT64はパケットのIPv4ヘッダをIPv6ヘッダに付け替え、
   元のIPv4-IPv6変換アドレスを送信元として通信を開始した端末へ返送する
5. CLATがIPv6パケットをIPv4パケットに変換し、アプリケーションへ渡す

## ipv4only.arpa
- DNSにドメイン名"ipv4only.arpa"のAAAAを問い合わせてDNS64が合成したIPv6アドレスを取得することにより、
  そのネットワークで使用されているNAT64 prefixを推定することができる
- 取得したIPv6アドレスから既知のIPv4アドレスを除いた部分がNAT64 prefix

```text
既知のIPv4アドレス

ipv4only.arpa. IN A 192.0.0.170
ipv4only.arpa. IN A 192.0.0.171
```

- Well-Known Prefix: 64:ff9b::/96 (そのネットワークのNAT64が64:ff9b::/96宛のパケットを受け付けている場合のみ有効)

#### 動作フロー
1. DNSにipv4only.arpaのAAAAレコードを問い合わせる
2. DNS64が192.0.0.170 / 192.0.0.171をNAT64 prefix付きでIPv6アドレスに合成してAAAAレコードとして返す
3. クライアントはAAAA応答からNAT64 prefixを推定
4. クライアントは任意のIPv4アドレスをNAT prefixに埋め込んでIPv6アドレスを合成
5. クライアントがIPv6アドレスへ接続すると、NAT64がIPv4アドレス宛に変換してオリジンサーバへ送信する

#### 関連RFC
- RFC 6052: IPv4アドレスをIPv6アドレスに埋め込むフォーマットを定義
- RFC 6146: NAT64 (IPv6クライアントからIPv4サーバへのプロトコル変換) を定義
- RFC 6147: DNS64 (AレコードからAAAAレコードを合成するDNSの動作) を定義
- RFC 7050: `ipv4only.arpa`を利用してNAT64 prefixを発見する方法を定義
  - RFC 8880: RFC 7050を更新 (`ipv4only.arpa`をSpecial-Use Domain Nameとして明確化)
- RFC 8781: RAのPREF64オプションでNAT64 prefixを通知する方法を定義
  - RFC 9872: NAT64 prefix discoveryの推奨事項を整理し、RFC 8781のRA PREF64の使用を推奨

## トンネリング
- あるプロトコルのパケットを別のプロトコルのパケットにカプセル化して転送する
  - 6in4 / 6rd: IPv6パケットをIPv4パケットにカプセル化して転送する仕組み (減少傾向)
  - DS-Lite: IPv4パケットをIPv6パケットにカプセル化して転送し、出口のCGNATでIPv4アドレスを共有する
  - MAP-E: IPv4パケットをIPv6パケットにカプセル化、数式的なルールでIPv4アドレスとポート番号をユーザに割り当てる
  - MAP-T:MAP-Eのカプセル化をトランスレーションに置き換えたもの
- 既存インフラを大きく変えずにIPv6を導入できる
- カプセル化によるオーバーヘッドがある、方式が多岐にわたり互換性がない

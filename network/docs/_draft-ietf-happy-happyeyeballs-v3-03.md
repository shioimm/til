# Happy Eyeballs Version 3: Better Connectivity Using Concurrency
https://datatracker.ietf.org/doc/draft-ietf-happy-happyeyeballs-v3/

## 1. Introduction
- 現代のインターネット上で動作する多くの通信プロトコルはホスト名を利用する
- ホスト名は多くの場合複数のIPアドレスに名前解決され、
  それぞれのIPアドレスが異なるパフォーマンスや接続特性を持つ場合がある
- 特定のIPアドレスやアドレスファミリは、あるネットワーク上で遮断されていたり、破損したり、
  最適でない状態になる可能性がある
  - そのため、複数の接続を並行して試行するクライアントは、より迅速に接続を確立できる可能性がある
- 本書では、デュアルスタックホストにおいてユーザが認識する遅延を低減するための技術である
  Happy Eyeballsアルゴリズムを定義する
- Happy Eyeballsアルゴリズムは、名前解決されたアドレス群に対する接続試行を競争的に行う方式
  - IPv6の使用やHTTP/3 [HTTP3]、TLS Encrypted Client Hello [ECH] などのプロトコルの利用可能性など
    クライアントの優先順位を尊重しつつ、可能な限りユーザへの遅延を回避するために、いくつかの段階で構成される
  - 本書では接続開始時にDNS問い合わせをどのように開始するか、
    DNS応答で得られた宛先アドレス一覧をどのように並べ替えるか、
    接続試行をどのように競争させるかについて論じる
- HEV2との主な違いは、SVCB / HTTPS Resource Records [SVCB] への対応が追加された点
  - SVCBレコードは代替エンドポイント、アプリケーションプロトコルの対応状況、Encrypted Client Hello (ECH) 鍵、
    アドレスヒント、アクセス対象サービスに関するその他の関連情報を提供する
  - 名前解決中にHTTP/3 over QUIC [HTTP3] などのプロトコルの対応状況を検出することにより、
    HTTP Alternative Services [AltSvc]などの他の検出メカニズムからの情報を使用して後続の試行を行うのではなく、
    現在進行中の接続試行の段階でプロトコルをアップグレードできる
  - これらのレコードはA / AAAAレコードとあわせて問い合わせすることができ、
    更新されたアルゴリズムでは接続確立を改善するためにSVCB応答をどのように扱うかを定義する

## 2. Conventions and Definitions
- MUST / REQUIRED / SHALL しなければならない
- MUST NOT / SHALL NOT してはならない
- SHOULD / RECOMMENDED する必要がある
- SHOULD NOT / NOT RECOMMENDED しないほうがよい
- MAY / OPTIONAL してもよい

## 3. Overview
- 本書は、Happy Eyeballs Connection Setupと呼ばれる接続確立方式を定義する

1. ホスト名を宛先アドレスへ非同期に名前解決する
2. 名前解決された宛先アドレスを並べ替える
3. 接続試行を非同期に開始する
4. 1つの接続を正常に確立し、その他の試行をキャンセルする

- 本書では、ホストの宛先アドレスの優先ポリシーが、IPv4よりもIPv6を優先することを前提とする
- 本書では、優先ポリシーがTCPよりもQUICを優先することを前提とする
- ホストが異なる優先設定となっている場合でも、本書の推奨事項は容易に適応できる

## 4. Hostname Resolution
- クライアントが名前付きホストへの接続を確立しようとする際、
  ホストに到達するために利用可能な宛先IPアドレスを特定する必要がある
- クライアントはDNS問い合わせを送信し、その応答を収集することで、ホスト名をIPアドレスへ名前解決する
- 本節ではクライアントがどのようにDNS問い合わせを開始し、その応答を非同期に処理するかについて論じる

### 4.1. Sending DNS Queries
- クライアントはまず、名前付きホストについてのDNS問い合わせに、どのDNSリソースレコードを含めるかを決定する必要がある
  - この決定は、クライアントがIPv4 / IPv6両方の接続性を有しているかどうかに基づく
    - 「接続性」とは、パケットを送信するためのファミリ内のローカルアドレスが少なくとも1つ、
      およびアドレスファミリー内の非リンクローカルルートが少なくとも1つ存在することを意味する

```text
(shioimm)

「接続性を有しているか」 = インターネットに接続できるアドレスを持っているかどうか

- パケットを送信するためのアドレスファミリ内のローカルアドレスを1つ以上持っていること
  (IPv4 / IPv6それぞれのIPアドレスが自らのNICに1つ以上割り当てられている)
- かつそれぞれのアドレスファミリ内の非リンクローカルルートが1つ以上存在すること
  (リンクローカルルート   = 同一セグメント内のみで有効なルート)
  (非リンクローカルルート = デフォルトルート 0.0.0.0/0や::/0など、外部ネットワークに到達できるルート)
```

- クライアントがIPv4とIPv6の両方の接続性を持つ場合、AAAA / Aレコード両方の問い合わせを送信する必要がある
  - IPv4接続性のみを持つネットワークでは、Aレコードへの問い合わせを送信する
  - IPv6接続性のみを持つネットワークでは、ネットワーク構成に応じてAAAA / A両方あるいはAAAAレコードのみ問い合わせる
    - (shioimm) PREF64あり -> AAAAとA両方を問い合わせる / DNS64のみ -> AAAAのみ問い合わせる
    - IPv6が主流もしくはIPv6のみのネットワークの扱いについては8. Supporting IPv6-Mostly and IPv6-Only Networksを参照
- 接続を確立するアプリケーションに応じて、クライアントはAAAA / Aレコードに加え、
  SVCBレコードまたはHTTPSレコードを問い合わせる場合があります [SVCB]
  - [SHOULD] HTTPまたはHTTPSを利用するアプリケーションの場合、クライアントはHTTPSレコードを問い合わせるべき
- [SHOULD] すべてのDNS問い合わせは、可能な限り連続して送信されるべき
- [SHOULD] 問い合わせの送信順序は、上記の条件により不要なものを省いたうえで、以下の順序とするべき:
  - 1. SVCBまたはHTTPSクエリ
  - 2. AAAAクエリ
  - 3. Aクエリ

### 4.2. Handling DNS Answers Asynchronously
- クライアントは、DNS問い合わせに対して十分な応答を受信した時点で、
  アドレスの並べ替えおよび接続確立の段階へ進むことができる
- [SHOULD NOT] 実装はすべての応答を待ってから次の接続確立手順を開始するべきではない
  - ある問い合わせが失敗したり、応答に著しく時間がかかる場合、それらの応答を待つことで、
    既に受信した応答に基づいて処理を進められるはずの接続確立が大幅に遅延する可能性があるため
  - [SHOULD] したがって、クライアントはDNS名前解決を非同期処理として扱い、異なるレコードタイプを個別に処理するべき
    - プラットフォームが非同期DNS APIを提供していない場合、
      各レコードタイプに対して個別の同期クエリを並列に実行することにより、この動作をシミュレートできる
- クライアントは、以下のいずれかの条件群を満たした時点で、アドレスの並べ替えおよび接続確立へ進む
  - 条件A:
    - 何らかの肯定的 (空でない) アドレス応答を受信した
    - かつ、問い合わせた優先アドレスファミリについて肯定的 (空でない) または否定的 (空の) 応答を受信した
    - かつ、SVCB / HTTPS のサービス情報を受信した (または否定的応答を受信した)
  - 条件B:
    - 何らかの肯定的 (空でない) アドレス応答を受信した
    - かつ、他の応答が届かないまま名前解決待機時間が経過した

```text
* 肯定的応答
  * AAAAレコードまたはAレコードに対する応答としてアドレスを含むもの
  * あるいはSVCB / HTTPSレコードに対する応答としてアドレスヒントを含むもの

* 否定的応答
  * AAAA / A レコードに対する応答で、アドレスを含まないもの
    * NXDOMAIN (DNSサーバが検索されたドメイン名が存在しない) のようなエラーを伴う場合も、伴わない場合も含む
    * すべての応答が否定的であった場合、接続確立は失敗するか、他の応答を受信するまで待機する必要がある
```

- 優先アドレスファミリ:
  - IPv6 / IPv4両方にデフォルトルートが存在するネットワークでは、IPv6が優先アドレスファミリであるとみなす
  - IPv6 / IPv4どちらか片方にのみデフォルトルートが存在するネットワークでは、そのアドレスファミリを
    アルゴリズム進行上の優先アドレスファミリとみなす

- Resolution Delay:
  - 優先アドレス (AAAAレコード経由) やサービス情報 (SVCB / HTTPS レコード経由) を受信する機会を得るため、
    短い待機時間を待つことがある (Resolution Delay)
  - AAAAレコードまたはSVCB / HTTPSレコードがAレコードより遅れて到着する可能性を考慮したもの
  - 推奨値は50ミリ秒

#### 4.2.1. Resolving SVCB/HTTPS Aliases and Targets
- SVCB / HTTPSレコードは、ネットワークサービスに関する情報を記述する
  - 個々のレコードはAliasModeまたはServiceModeのいずれか
    - AliasModeの場合、TargetNameは別名先であり、エイリアス名を取得するために追加のSVCB / HTTPS問い合わせが必要
    - ServiceModeの場合、TargetNameは`.` (問い合わせ対象元) または別ホストであり、追加のSVCB / HTTPS問い合わせは不要
    - HEv3では、ServiceModeレコードが利用可能になるまで、サービス情報を受信したとはみなさない
  - ServiceModeレコードは`ipv6hint`および`ipv4hint`パラメータを介してアドレスヒントを含めることができる
    - [SHOULD] これらのヒントを受信した時点で、TargetNameに対するA / AAAAレコードがまだ利用可能でない場合は、
      アルゴリズム上、このヒントを肯定的かつ空でない応答として扱うべき
      - shioimm: 利用可能でない = 応答を受信していない
      - ただしTargetNameに対するA / AAAA問い合わせの応答を受信していない限り、
        クライアントは引き続きA / AAAA問い合わせを発行する必要がある
        - 実際のA / AAAA問い合わせに対する応答を受信した時点で、実際のレコードがヒントを置き換え、
          利用可能な応答セットが新たな応答として更新される

### 4.3. Handling New Answers
- 接続試行中、接続確立前に新しいレコードが到着した場合、
  新しく受信したアドレスは利用可能な候補アドレスのリストに追加される
- 接続が確立されるまで、追加された新しいアドレスも含めて接続試行プロセスを継続する

### 4.4. Handling Multiple DNS Server Addresses
- (DNSサーバ自身のアドレスとDNSサーバに対するDNSメッセージの転送に使用されるIPバージョンについて)
  クライアントの現在のネットワークに複数のDNSサーバアドレスが設定されている場合、
  クライアントはDNS問い合わせをIPv4 / IPv6どちらで送信するかを選択できる
- [SHOULD] Happy Eyeballsの考え方に従い、問い合わせはまずIPv6上で送信するべき
  - DNSサーバのIPv6アドレス宛に送信したDNS問い合わせに対して、応答を受信できない場合、
    そのアドレスをペナルティ対象としてマークし、他のDNSサーバアドレスへ問い合わせを送信できる
  - ネイティブIPv6の導入が進み、IPv4アドレスが枯渇していくにつれ、
    ネットワーク内ではIPv6接続性が優先的に扱われることが想定される
  - DNSサーバがIPv6経由で到達可能なよう設定されている場合、IPv6を優先アドレスファミリとみなすべき
- [SHOULD NOT] クライアントシステムは、手動またはネットワークによって設定できるDNSサーバーの数に
  明示的な上限を設けるべきではない
  - [SHOULD] ハードウェア上の制約によりそのような上限が必要な場合な場合
    クライアントは利用可能な一覧から、各アドレスファミリごとに少なくとも1つのアドレスを使用する必要がある

## 5. Grouping and Sorting Addresses
- 名前解決された宛先アドレスのいずれかに接続を試みる前に、クライアントは接続試行を開始する順序を決定する
- 順序が定義された後、クライアントは短い遅延を挟みながら各候補を競争的に試行する単純なアルゴリズムを使用できる
- 順序付けされたリストには、この時点で受信したすべてのIPv4アドレスとIPv6アドレス、
  および両方のファミリのすべてのアドレスとすべてのプロトコルが含まれている必要がある
  - これによってクライアントは、最初のIPv4アドレスと最初のIPv6アドレスだけでなく、
    リスト全体に対してHappy Eyeballsの競争効果を得ることができる
- クライアントは受信したDNS応答に基づき、アドレスに対して3段階のグループ化と並べ替えを行う
  - 1. アプリケーションプロトコルおよびセキュリティ要件によるグループ化・並べ替え (5.1)
  - 2. サービス優先度によるグループ化・並べ替え (5.2)
  - 3. 宛先アドレスの優先設定による並べ替え (5.3)

### (shioimm)
1. アプリケーションプロトコルおよびセキュリティ要件によるグループ化・並べ替え (5.1)
    - アプリケーションにとって本当に意味のある差異があるときだけグループを分ける必要がある
    - グループ自体のソートは実装依存
2. サービス優先度によるグループ化・並べ替え (5.2)
    - SVCBがある場合: サービスごとにグループ化し、優先度の数値が小さい順に並べる
    - 同優先度が複数ある場合: ランダムシャッフル
    - SVCBがない / 全部同じ場合: 全部1グループ
    - 紐付かないA / AAAAの場合: 末尾グループ
3. 宛先アドレスの優先設定による並べ替え (5.3)
    - グループ内での並び替えルールはv2と同じ

### 5.1. Grouping By Application Protocols and Security Requirements
- クライアントはまず、宛先エンドポイントがサポートするアプリケーションプロトコルと、
  それらのエンドポイントが提供するセキュリティ機能に基づいてグループ化を行う
  - これらの情報はSVCB / HTTPSレコードに含まれるアプリケーション層プロトコル (`alpn`値) や、
    TLS Encrypted Client Hello 設定 (`ech`値 [SVCB-ECH]) などの各種パラメータに基づく
    - 応答にSVCB / HTTPS情報が含まれない場合、またはすべての応答が同一のSVCB / HTTPSレコードに対応している場合:
      - すべての応答は1つのグループに属し、クライアントはそれらが同じプロトコル・同じセキュリティ特性を持つとみなす
    - 複数の異なるSVCB / HTTPSレコードを受信した (= 異なる能力を広告する複数の宛先エンドポイントがある) 場合:
      - [SHOULD] クライアントは、各グループ内のすべてのアドレスが
        同じアプリケーションプロトコルと関連するセキュリティ特性を共有するように、
        これらのアドレスを異なるグループに分割する
- クライアントにとって重要なパラメータはクライアント実装およびアプリケーションによって異なる
  - (shioimm) SVCBレコードに含まれる`alpn`、`ech`、`port`などのパラメータのうち、
    どれをグループ分けの基準として用いるかはクライアント実装に委ねられる
    - デフォルトではサーバの優先度に従い、オプションで上書きできるようにするとか?

```text
一部の宛先アドレスは複数のグループに追加する必要がある場合もある

e.g.
example.com. 60 IN HTTPS 1 svc1.example.com. (alpn="h3,h2" ipv6hint=2001:db8::2)
example.com. 60 IN HTTPS 1 svc2.example.com. (alpn="h2"    ipv6hint=2001:db8::4)

- 2001:db8::2はHTTP/3とHTTP/2の両方に使用できる
- 2001:db8::4はHTTP/2にのみ使用できる

クライアントが HTTP/3対応アドレス群A と HTTP/2対応アドレス群B でグループ化する場合、
2001:db8::2はA/B両方のグループに含まれる (他のセキュリティ特性がすべて同じであると仮定した場合)
```

- 接続試行時の競合は、各グループ内での宛先アドレス候補に対して順に適用される
  - 異なるセキュリティ特性やプロトコル特性を持つグループ間での優先順位付けやフォールバックの方法は実装依存

#### 5.1.1. When to Apply Application Preferences (グループ化を適用するべき条件)
- 特定のアプリケーションプロトコルやセキュリティ機能を個別にグループ化するかどうかは、
  クライアントアプリケーションの判断に委ねられる
  - [SHOULD] クライアントにとってそのアプリケーションプロトコルや機能を使用することが重要でない場合、
    別個にグループ化・並べ替えを行うことは避けるべき
  - e.g.
    - 単純なWebページを読み込むHTTPクライアントの場合:
      - HTTP/3とHTTP/2のどちらを使ってもパフォーマンスの大きな違いはない可能性がある
        - -> サービス側で定められた優先順位 (HTTP/3がHTTP/2より優先順位が低いなど) を尊重し、ALPNをグループ化する
    - HTTP/3が持つunreliable frames送信機能によりアプリケーション用途で大きな性能向上が得られるクライアントの場合:
      - -> HTTP/3をHTTP/2より優先する形で別グループ化する
    - プライバシーに敏感な通信を行うクライアントの場合:
      - -> TLS ECHの利用を必須または強く優先する
    - プライバシーに敏感な通信を行わないクライアントの場合:
      - -> ECHは機会的に利用するだけなのでグループ化を行う際には考慮しない

- ECHを含む応答のグループ化:
  - [SVCB-ECH] 8では、ECH構成を含む応答と含まない応答が混在するSVCBレコードセットの使用を推奨していないものの、
    そのようなケースも起こり得る
    - サービスによっては、ECH構成を含まない応答よりも優先順位の低いSVCB応答にのみECH構成を含める場合がある
      (実験導入や段階的ロールアウト戦略のため)
    - このようなケースがあるため、
      クライアントはECH情報を使用しない場合または接続にECHの使用によるメリットがない場合、
      ECHを含む応答を恣意的にグループ化して優先的に並べ替えるべきではない
      - [MAY] アプリケーション側がECHを優先する合理的理由がある場合、
        クライアントはそれらの応答を個別にグループ化して優先してもよい
        - これは公開されているサービスレコードの優先順位と矛盾する可能性があるが、
          サービスによって公開された応答はすべてクライアントが使用することができ、
          クライアントはそれらを使用するかどうかを選択できる

### 5.2. Grouping By Service Priority
- アプリケーションプロトコルやセキュリティ要件によるアドレスのグループ化 (5.1) を行った後、
  次のステップではSVCB / HTTPSレコードで定義された異なるサービス間でグループ化を行い、
  これらのグループを優先度順に並び替える
  - このステップにより、サーバが公開する優先度をクライアントの接続確立アルゴリズムに反映させることができる
- SVCB [SVCB] レコードは、各ServiceMode応答に対して優先度を示す
  - この優先度は、レコード自体に含まれるIPv4またはIPv6アドレスヒント、
    ServiceModeレコード内の名前に対するAまたはAAAAクエリで受信したすべてのアドレスに適用される
    - SVCB ServiceModeレコードの優先度は常に0より大きい
    - 数値として最も小さいSVCB応答 (1など) を先に並べ、より大きい数値の応答を後ろに並べる
  - TargetNameが`.`のSVCBレコードはそのレコードのowner nameに適用され、
    そのSVCBレコードの優先度は、同じowner nameに対するすべてのA / AAAAレコードに適用される
    - これらの応答は、そのSVCBレコードの優先度に従って並べ替えられる
  - [SHOULD] 特定のSVCBサービスに関連付けられたAAAA / Aレコード、アドレスヒントによって受信したすべてのアドレスは、
    クライアントによってグループに分割されるべき
    - [SHOULD] これらのサービスベースのグループは、サービスの優先度に基づいてソートされるべき
- 応答にSVCB / HTTPS情報が含まれない場合、あるいはすべての応答が同一のSVCB / HTTPSレコードに対応している場合:
  - すべての応答は同じ優先度を持つ1つのグループに属する
- [SHOULD] 同じ優先度を持つサービス、複数のグループが存在する場合:
  - クライアントはこれらのグループをランダムにシャッフルするべき
- [SHOULD] 一部の受信済みSVCB / HTTPS サービス情報に、
  どのSVCBにも紐付かないAAAA / Aレコードが存在する場合:
  (e.g. `.`で始まる元の名前に対してSVCB / HTTPSレコードが受信されない場合など)
  関連付けられていないアドレスは、リストの末尾に末尾に低優先度グループとして並べるべき

```text
(shioimm)
example.com. IN HTTPS 1 svc1.example.com. (alpn="h3,h2")
example.com. IN HTTPS 2 svc2.example.com. (alpn="h2")

svc1.example.com. IN AAAA 2001:db8::1   <- svc1に紐付いたAAAAレコード (SVCBの優先度でグループ化)
svc2.example.com. IN AAAA 2001:db8::2   <- svc2に紐付いたAAAAレコード (SVCBの優先度でグループ化)
example.com.      IN AAAA 2001:db8::99  <- どのSVCBにも紐付かないAAAAレコード (末尾に低優先度グループとして追加)
example.com.      IN A    192.0.2.1     <- どのSVCBにも紐付かないAレコード (末尾に低優先度グループとして追加)
```

### 5.3. Sorting Destination Addresses Within Groups
- サービスに基づくグループ化 (5.2) を行った後、
  各アドレスグループ内で、クライアントは優先度および過去の履歴データに基づいてアドレスを並べ替える

- 1. 履歴による並び替え;
  - [MUST] クライアントは宛先アドレス選択 ([RFC 6724] 6) を用いてアドレスを並べ替えなければならない
    - [SHOULD] クライアントが状態を保持しており、
      各アドレスへ到達する経路ごとの想定往復時間 (RTT) の履歴を持っている場合、
      ルール8とルール9の間に、RTTが低いアドレスを優先するルールを追加するべき
    - [SHOULD] クライアントが過去に使用したアドレスを記録している場合、
      RTTルールとルール9の間に、未使用のアドレスより使用済みアドレスを優先する別の宛先選択ルールを追加するべき
      - これはTCP Fast Open ([RFC 7413]) や一部のHTTP Cookieのように、
        認証時にクライアントのIPアドレスを認証に利用するサーバに有効
      - [MUST] 履歴データは、当該エンドポイントに固有のプライバシーに関わる機密情報に使用されるのと
        同じ境界で分離しなければならない
        - [MUST NOT] 異なるネットワークインターフェース間で共有してはならない
        - [SHOULD] 端末が接続先ネットワークを変更した場合、このデータは消去されるべき
          - [MAY] クライアントが以前接続していたネットワークを確実に識別できる場合、
            再接続時にそのネットワークに関連づけられた履歴データを保持し、再利用してもよい
        - [MUST] 履歴データを利用するクライアントは、異なる履歴を持つクライアント同士が
          最終的には同様の挙動へ収束することを保証しなければならない
          - e.g. クライアントは定期的に履歴データを無視することにより、新しいアドレスへ接続試行するようにできる

- 2. アドレスファミリによる並び替え:
  - [SHOULD] クライアントは、アドレスファミリが交互になるように並び順を調整するべき
    - リストの先頭にあるアドレスファミリの後に、もう一方のアドレスファミリのエンドポイントが続くべき
      - e.g. 並べ替え後のリストの先頭がIPv6 アドレスである場合、2番目のアドレスはIPv4アドレスであるべき
      - [MAY] 実装によっては特定のアドレスファミリをより優先的に使用するように選択し
        そのアドレスファミリの複数アドレスを連続して試行してから、次のアドレスを試行してもよい
      - 先頭アドレスファミリに属する連続アドレス数はPreferred Address Family Countと呼ばれ、設定可能な値とできる
        - これにより、特定のアドレスファミリの接続が障害を受けた場合に、
          そのアドレスファミリに属するアドレスの長いリストを順番に試行する必要がなくなる

- 本節で述べるアドレス選択は、宛先アドレスのみに適用される
  - 送信元アドレス選択 (Source Address Selection) は各宛先アドレスごとに実行されるものであり、
    本書の対象範囲外 ([RFC 6724 Update])

## 6. Connection Attempts
- グルーピング・並び替えされたアドレス一覧を構築したあと、クライアントは接続試行を開始する
  - [SHOULD NOT] 過度なネットワーク負荷を避けるため、接続試行は同時に開始するべきではない
    - 代わりにまず1つのアドレスに対する接続試行を開始し、その後、他のアドレスに対して順に1つずつ試行を開始します
- 新たな接続試行の開始は、それ以前の試行に影響を与えない
  - 複数の接続試行は並行して進行する場合がある
  - [SHOULD] いずれかの接続試行が成功した時点で、まだ成功していない他の接続試行はキャンセルされるべき
    - [SHOULD] その際、まだ試行されていないアドレスは無視されるべき
- [MAY] 進行中の非同期DNS問い合わせはキャンセルしてもよい
  - [SHOULD] DNSクライアントリゾルバはネットワークからのDNS応答を短時間 (推奨値は約1秒) 処理し続けるべき
    - これによりDNSキャッシュが更新され、後続の接続に利用できる
- [SHOULD] グルーピングの結果、複数のグループが生成された場合、
  クライアントは最も優先度の高いグループから接続試行を開始するべき
  - それ以外のグループに対する試行ポリシーはクライアント実装に依存する
- [SHOULD] グルーピングの結果、複数のグループが生成された場合、
  次のグループに進む前に最初のグループ内のすべてのアドレスに対する試行を開始するべき
  - [SHOULD] サービスグループ間の試行は並行して継続されるべき
    - 実質的にはすべてのグループが1つのリストにフラット化された形になる
- 単純な実装では、次の接続試行を開始するまでの待機時間として固定値 Connection Attempt Delay を使用でき、
  既定値としては250ミリ秒が推奨される
  - より高度な実装ではこの遅延は、前回の接続試行がハンドシェイクの再送
    (たとえば2回目の TCP SYN や QUIC Initial) を行うタイミングに対応させるべき
    - これは再送タイマー ([RFC 6298]、[RFC 9002]) に基づく
- 同一ホストまたは同一プレフィックスへの過去の接続から得られたRTTデータがある場合、
  クライアントはそれを遅延値の決定に利用できる
  - このアルゴリズムは最初の再送タイミングを近似することのみを目的とし、
    指数バックオフの影響を受けるその後の再送までは考慮しない
- [SHOULD] TLSセッションチケット (TLS 1.3) やQUIC 0-RTT (QUIC TLS) などのセッション再開機構をサポートする
  クライアントは新規接続と同様に、再開接続に対しても同じ Connection Attempt Delay を使用するべき
- [MUST] Connection Attempt Delayには下限が必要であり、特に履歴データに基づいて計算される場合は重要
  - [MUST NOT] 具体的には、前回の試行から10ミリ秒未満で次の接続試行を開始してはならない
  - Minimum Connection Attempt Delay: 推奨される最小値は100ミリ秒
  - この最小値は、高いパケットロス環境において輻輳崩壊を防ぐために必要
- [SHOULD] Connection Attempt Delayには上限も設けるべき
  - Maximum Connection Attempt Delay: 現在の推奨値は2秒
- Connection Attempt Delayは、Next Connection Attempt Timerと呼ばれるタイマーの設定に使用される
  - このタイマーが発火した時点で接続がまだ確立していなければ、次の接続試行が開始される
  - その後タイマーは新しい遅延値で再設定されるか、リスト末尾に達した場合はキャンセルされる
  - この遅延値は、使用されるプロトコルや推定RTTに応じて、接続試行ごとに異なってもよい

### 6.1. Determining successful connection establishment
- 接続試行が成功したとみなされるタイミング、および他の試行をキャンセルできるタイミングの判定は、
  最終的にはクライアントアプリケーションが「その接続が利用可能な状態になった」と解釈する基準に依存する
  - 一般にはリモートエンドポイントとのトランスポート層のハンドシェイク (TCP / QUIC) の完了が含まれるが、
    それに加えて、より上位レベルのハンドシェイクや状態確認が含まれる場合もある
    - TCPのみを使用する接続 (TLSなどの上位プロトコルを使用しない e.g. 非暗号化HTTP) では、
      TCPハンドシェイクの完了をもって接続成功と判断する
    - [SHOULD] TCP上でTLS を使用する場合 (e.g. HTTPS接続) 、
      クライアントはTLSハンドシェイクが正常に完了するまで待ってから他の接続試行をキャンセルするべき
      - TCPハンドシェイクは迅速に成功するものの、
        TLS終端サーバとのエンドツーエンド接続が失敗するようなTCP終端プロキシが存在するネットワークにおいて有効
    - QUIC接続は、そのハンドシェイク自体にセキュアなハンドシェイクが含まれているため、
      通常は単一のハンドシェイクの完了を待つだけでよい
    - クライアントは接続確立の判断に際し、TCP / TLS / QUICハンドシェイク以外の要件を満たすまで待機する場合もある
      - e.g. TLSにより提供されるサーバ証明書が信頼できることの検証など
      - 接続確立の判定が初期のトランスポートハンドシェイクを超える場合、
        初期ハンドシェイク完了後にNext Connection Attempt Timerを調整するべき
        - [SHOULD] 接続確立が進行しているが未完了である場合、タイマーは完全な接続確立が完了すると見込まれる時間を
          反映した新しい値に延長されるべき
        - e.g. 接続確立にTCPハンドシェイクとTLSハンドシェイクの両方が含まれる場合:
          - タイマーがTCP SYNの再送が行われるタイミングに近い値に初期設定されており、
            そのタイマー発火前にTCPハンドシェイクが完了した場合、
            TLSハンドシェイク完了に必要な時間を考慮し、タイマーを調整する
- トランスポート層のハンドシェイクでは通常接、続試行回数に関する制約はないが、
  一部の暗号ハンドシェイクはSVCB ServiceModeレコードに依存し、接続確立に制約を課す場合がある
  - ECH対応クライアントは、SVCBレコードに`"ech"` SvcParamKey ([[SVCB-ECH]]) が含まれる場合、
    SVCB-reliantクライアントとなる可能性がある ([SVCB])
    - [MUST] クライアントがSVCB-reliant、
      または (処理中にSVCB依存接続確立に切り替える可能性がある) SVCB-optionalである場合、
      クライアントは暗号化ハンドシェイクを開始する前にSVCBレコードを待機する必要がある

#### (shioimm)
- SVCBレコードに"ech"キーが含まれている場合、SVCB-reliantになる可能性がある
  - SVCB-reliant: SVCBレコードが取得できなければ接続しない
  - SVCB-optional: SVCBレコードがなくても接続できる (ECHを使うことができれば使う)
- SvcParamKey: SVCB / HTTPSレコードに記述されるサービスパラメータ群
  - alpn: 対応するアプリケーション層プロトコル (e.g. `h2=HTTP/2`、`h3=HTTP/3`)
  - no-default-alpn: デフォルトのプロトコルネゴシエーションを無効化する
  - port: 接続先ポート番号
  - ipv4hint: IPv4アドレスのヒント (DNS解決を省略するためのキャッシュ)
  - ipv6hint: IPv6アドレスのヒント
  - ech: ECH (Encrypted Client Hello) の公開鍵設定

### 6.2. Handling Application Layer Protocol Negotiation (ALPN)
- SVCBレコードに含まれる`alpn`および`no-default-alpn`のSvcParamKeyはSVCB ALPNセットを示し、
  対応するサービスエンドポイントがサポートする下位トランスポートプロトコルを指定する
- [SHOULD] クライアントがSVCBレコードを要求する場合、
  クライアントとサービスエンドポイントの双方がサポートするトランスポートプロトコルを決定するため
  SVCB / HTTPS Resource Recordsで規定された手順を実行するべき
- [SHOULD NOT] サービスエンドポイントの持つSVCB ALPNセットに、
  クライアント自身のサポートするプロトコルが1つも含まれていない場合、
  クライアントはそのサービスエンドポイントに対して接続を試みるべきではない

```text
e.g.
HTTP/1.1やHTTP/2のようなTCPベースのプロトコルのみをサポートするHTTPクライアントが、
次のようなHTTPSレコードを受信した場合:

example.com. 60 IN HTTPS 1 svc1.example.com. (alpn="h3" no-default-alpn ipv6hint=2001:db8::2)

このレコードは、ALPN 値"h3"により、svc1.example.comがHTTP/3のみをサポートしていることを示す
そのため、このクライアントが2001:db8::2やsvc1.example.comに対応する他のアドレスへ接続を試みることは不適切
```

- [SHOULD] HTTP クライアントがAlt-SvcとSVCB (HTTPS) レコードの両方をサポートしている場合、
  接続試行はAlt-SvcパラメータとSVCB ALPNセットの両方と整合するように行うべき

### 6.3. Dropping or Pending Connection Attempts (接続試行の中止および保留)
- SVCB応答の扱いに関連するいくつかの状況では接続試行を中止する、あるいはSVCB応答が返るまで保留する必要がある
- (接続試行の中止) DNSSECや暗号化DNS (DoT / DoH / DoQ) を使っている環境で、SVCBレコードの名前解決が失敗した場合
  - SVCB / HTTPS Resource RecordsではDNSSEC、DNS over TLS (DOT) 、DNS over HTTPS (DOH) 、DNS over QUIC (DOQ) などの
    暗号学的に保護されたDNSを使用している場合の名前解決失敗時のクライアント挙動が説明されている
    - [SHOULD] このような環境でSVCB解決に失敗した場合、ダウングレード攻撃を防ぐため、
      クライアントは接続試行自体を中止するべき
    - DNSにおける暗号学的保護の有無は、Happy Eyeballsによる接続確立の他の部分にも影響を与える
      - DNSが保護されていない場合、レコードはブロックまたは改ざんされ得るため、
        SVCB レコードから得られるセキュリティ特性はあくまで機会的なものにとどまる
      - DNSが暗号学的に保護されている場合、クライアントはSVCBレコードに基づく特性に対し、より厳密に依存できる
- (接続試行の保留) SVCBレコードが未受信、かつTLSハンドシェイクを始めようとしている場合
  - SVCB / HTTPS Resource Recordsは
    「[MUST NOT] クライアントはSVCB応答が到着するまで、その応答によって変更され得る情報を送信してはならない」
    と説明されており、特にTLS ClientHelloに影響を与える特性が言及されている
    - この制約はSVCB応答によってクライアントの挙動が変更される場合に適用される
    - これはクライアント実装が特定の機能をサポートしているかどうか、
      およびその機能を有効化するためにSVCB応答に依存するかどうかの両方に依存します。
    - [MUST] これに基づき特定の状況では、クライアントはTLSハンドシェイクの開始 (TCP / QUIC) を
      SVCB応答を受信するまで保留しなければならない
      - これはResolution Delayを過ぎた後であっても適用される

以下のすべての条件が満たされる場合、ハンドシェイク開始を保留しなければならない

```text
1. DNS応答がDNSSECや暗号化DNSによって暗号学的に保護されている
   [MAY] 非暗号化・非署名DNSの場合、SVCB情報は機会的なものとなるため、クライアントは待機してもよい

2. クライアント実装が特定のセキュリティ関連SVCBパラメータ (e.g. "ech" SvcParamKey、SVCB-ECH) を解析・利用できる

3. クライアントがそのSVCBパラメータの存在に依存して、該当プロトコル機能を有効化する
   通常は平文HTTPを使用するが、HTTPSレコードによってアップグレードする場合、そのレコードを待つ必要がある
   一方、もともとTLS上のHTTPを使用する場合は、そのSVCB シグナルに依存していないため待機は不要
```

- 一部のSVCBプロパティはTLS ClientHelloに影響を与え、パフォーマンス最適化 (e.g. `tls-supported-groups`) を行う
  - これは往復回数削減を目的としたものであり、TLSハンドシェイク自体で必要な情報は取得可能であるため、
    このような場合にはSVCB応答を待機する必要はない

## 7. DNS Answer Changes During Happy Eyeballs Connection Setup
- 接続確立中、DNS応答が変化する場合がある
  - e.g. DNSプッシュ通知 (DNS Push Notifications) により新たなアドレスが追加される、
    TTLの期限切れにより既存のアドレスが削除される など
    - このような場合、クライアントは現在の進行状況に応じて適切に対応する必要がある
- [SHOULD] SVCBのIPヒントによって得られたアドレスについて、対応する名前に対するA / AAAAレコードを受信した時点で
  それらのレコードに含まれていない場合、一覧から削除するべき ([SVCB / HTTPS Resource Records])
  - [SHOULD NOT] 一覧から削除されるべきアドレスに対して接続試行がすでに開始されている場合、
    その接続試行はキャンセルせず、そのまま継続させるべき
  - [SHOULD] そのアドレスに対する接続試行がまだ開始されていない場合は、試行対象リストから削除するべき
- [SHOULD] 新たなアドレスが一覧に追加された場合、既存のアドレスも含めた全体に対して並べ替えルールを再適用するべき
  - これにより、アドレスファミリの交互配置などの並べ替えルールが、アドレスの到着タイミングに依存せず正しく維持される
  - e.g. 初期状態では IPv6アドレスのみが利用可能で、そのうちの1つに対する接続試行がすでに進行している場合:
    - その後IPv4アドレスが到着した場合、そのIPv4アドレスは、アドレスファミリの交互配置を考慮して、
      残りのIPv6 アドレスよりも前に配置され、次に試行されるべきアドレスとなる

## 8. Supporting IPv6-Mostly and IPv6-Only Networks
- 多くのIPv6移行プロトコルは標準化され展開されているが、その多くはクライアントデバイスからは透過的に扱われる
  - IPv6専用ネットワークを適切にサポートするために、特にIPv4のみをサポートするサービスと通信する場合、
    クライアント側での対応が必要になることがある
    - 代表的な方法としては、NAT64とDNS64を組み合わせて利用する方法、
      あるいはNAT64と検出されたPREF64プレフィックス (PREF64 Prefix Discovery) を組み合わせて利用する方法がある
      - これらのネットワークに対応する一つの方法として、
        クライアントデバイスのネットワークスタックが464XLATを実装することが挙げられる
        - 464XLATは、ユーザー空間ソフトウェアに変更を加える必要がないという利点があるが、
          アプリケーションがIPv4リテラルを使用する場合にはパケットごとの変換が必要となり、
          またクライアントアプリケーションがネイティブIPv6をサポートする方向を促進しにくいという側面もある
        - [SHOULD] 464XLATをサポートしていないプラットフォームにおいては、Happy Eyeballsの実装は、
          本節で示す推奨事項に従い、IPv6主体ネットワーク (IPv6-mostly network)
          およびIPv6専用ネットワークを適切にサポートするべき
        - [SHOULD] 本節で説明する機能はホストがIPv6主体ネットワークまたはIPv6専用ネットワークであると
          検出された場合にのみ有効化するべき
          - それらのネットワークを検出する簡易的な方法:
            - ルーティング可能なIPv6アドレスが提供されていること
            - ルーティング可能なIPv4アドレスが提供されていないこと
            - DNSリゾルバのアドレスが提供されていること

### 8.1. IPv4 Address Literals
- クライアントアプリケーションまたはユーザがIPv4アドレスリテラルへ接続しようとする場合、
  Happy Eyeballsエンジンは、そのためにNAT64アドレス合成を行う必要がある
  - この手法は、Bump-in-the-Hostと類似していますが、Happy Eyeballsクライアント内部で実装される
  - いくつかのIPv4プレフィックスは特定のホストまたはネットワークにスコープされておりNAT64アドレス合成を必要としない:
    - 0.0.0.0/8
    - 127.0.0.0/8
    - 169.254.0.0/16
    - 255.255.255.255/32

### 8.2. Discovering and Utilizing PREF64
- [SHOULD] ホスト名ではなくIPv4アドレスがHappy Eyeballs実装に渡された場合、
  ルータ広告で通知されるPREF64 (PREF64 Prefix Discovery) を使用するべき
  - PREF64が利用可能な場合、ネットワーク側はDNS64を展開しない選択をすることがある
    - DNS64にはいくつかの欠点があるため ([IPv6-mostly networks])
      - [SHOULD] このようなネットワークとの互換性を確保するため、PREF64が利用可能な場合、
        クライアントは特定のホスト名に対してAAAA問い合わせに加えてA問い合わせも送信するべき
        - これにより、既存のIPv4 Aレコードを取得し、クライアント側でNAT64アドレス合成を行うことができ、
          ネットワーク側でDNS64を動作させる必要がなくなる
  - [SHOULD] ネットワークがPREF64を提供していない場合、実装はNA64 Prefix Discovery
    (IPv6アドレス合成に使用されるIPv6 プレフィックスの検出) に従ってNAT64 プレフィックスを問い合わせるべき
    - そのうえでIPv6 Addressing of IPv4/IPv6 Translators (IPv4/IPv6トランスレータのためのIPv6アドレス方式) で
      定義されたエンコーディングを用いて、適切なIPv6アドレスを合成する
- これらの合成アドレスは、DNSのAレコード応答から得られたかのようにアドレス一覧へ挿入され、
  その後の接続試行は本書で説明したアルゴリズムに従って行われる
- 同様の変換は、Aレコードから取得したIPv4アドレスや、SVCBレコード内のIPv4 address hintsに対しても適用される

### 8.3. Supporting DNS64
- [SHOULD] PREF64が利用できず、かつNAT64プレフィックスも検出できない場合、
  クライアントはそのネットワークがIPv4からIPv6へのアドレス合成にDNS64を利用していると仮定するべき
  - この場合、DNS64サーバがIPv4のみのドメインに対してAAAAレコードを合成するため、
    クライアントは通常、DNS問い合わせの結果としてAAAAレコードのみを受信することになる

### 8.4. Hostnames with Broken AAAA Records
- 執筆時点において有効なAレコードと壊れたAAAAレコードを同時に持つホスト名が存在する
  - 壊れたAAAAレコード: 一見有効なIPv6アドレスを含んでいるものの、通常のポートで接続しても応答が返らないもの
- このような問題は、以下のような要因によって発生する可能性がある
  - DNS ゾーン設定における IPv6 アドレスの誤記
  - ルーティングブラックホール
  - サービス障害
- 本書の他の節に従ったアルゴリズムであれば、デュアルスタックネットワーク上ではこれらのホスト名にも正しく対応できるが
  NAT64とDNS64を用いるIPv6専用ネットワークでは、必ずしも正しく動作しない
  - DNS64の再帰リゾルバがAAAAレコードに対して否定応答を受け取ることを前提に合成を行うため
  - このようなホスト名では壊れたAAAAレコードが存在するため、DNS64は合成を行わず、そのまま壊れたAAAAレコードを返す
  - このようなケースに対応するため、クライアントはAレコードをDNSに問い合わせ、
    その結果に対してローカルでアドレス合成を行う必要がある
    - ただしこの種のホスト名は稀であり、DNSサーバへの負荷を抑えるためにも、
      このAレコード問い合わせは最初に取得したAAAAレコードを使った接続試行を断念した場合にのみ実行するべき
      - これを実現するため、Last Resort Local Synthesis Delayと呼ばれる長めのタイムアウト (推奨値は2秒) を使用する
        - このタイマーは最後の接続試行が開始された時点で起動される
        - このタイマーが発火しても接続試行が成功していない場合、
          デバイスはDNSに対してIPv4アドレス (Aレコード) を問い合わせる
          - 有効なAレコードを受信した場合、それをアプリケーションから直接渡されたものとして扱う

### 8.5. Virtual Private Networks
- 一部の仮想プライベートネットワーク (VPN) は、デバイスからのDNS問い合わせを処理するように構成されている
  - この構成はすべての問い合わせを対象とする場合もあれば、`*.internal.example.com` のような特定のサブセットのみを
    対象とする場合もある
  - これらのVPNは、IPv4アドレス空間の一部のみ (e.g. 192.0.2.0/24) をルーティングするように設定されることもある
  - 内部ホスト名が外部のIPv4アドレスへ名前解決される場合、
    基盤となるネットワークがIPv6専用であると問題が発生する可能性がある
    - e.g. `www.internal.example.com`がAレコードとして`198.51.100.42`のみを持ち、AAAAレコードを持たない場合:
      - クライアントはこのDNS 問い合わせを企業の再帰リゾルバへ送信し、そのリゾルバはこのレコードを返す
      - その結果、デバイスは接続先としてIPv4アドレスのみを持つことになりますが、そのアドレスへの経路は存在しない
  - 企業のリゾルバは基盤ネットワークのNAT64プレフィックスを知らないため、IPv6アドレスを合成できない
  - 同様に基盤ネットワーク側のDNS64再帰リゾルバは、企業内部のアドレス空間を知らないため、そのホスト名を解決できない
  - このためクライアントデバイスは、企業のリゾルバを用いてAレコードを解決し、
    その後得られたIPv4 アドレスをアプリケーションから直接渡されたものとして扱い、
    ローカルで IPv6 アドレス合成を行う必要がある

## 9. Summary of Configurable Values (クライアント上で既定値として設定可能な値)
- Resolution Delay
  - Aレコード受信後、AAAA and / or SVCB / HTTPSレコードを待機する時間
  - 推奨値は50ミリ秒
- Preferred Address Family Count
  - 優先アドレスファミリに属するアドレスを、次のアドレスファミリへ進む前に試行する件数を示す
  - 推奨値は1
  - 特定のアドレスファミリとプロトコルの組み合わせをより積極的に優遇したい場合は2を使用してもよい
- Connection Attempt Delay
  - RTTデータが存在しない場合に、接続試行間で待機する時間
  - 推奨値は250ミリ秒
- Minimum Connection Attempt Delay
  - 接続試行間で待機する最小時間
  - 推奨値は100ミリ秒
  - [MUST NOT] 10ミリ秒未満にしてはならない
- Maximum Connection Attempt Delay
  - 接続試行間で待機する最大時間
  - 推奨値は2秒
- Last Resort Local Synthesis Delay
  - 最後のIPv6接続試行を開始してから、Aレコード問い合わせを送信するまで待機する時間
  - 推奨値は2秒
- 本節で述べた遅延値は、非常に幅広い実運用デバイス群における接続タイミングを測定した経験的データに基づいて決定された
  - これらの値は、ユーザーが感じる待ち時間を減らしつつ、ネットワーク負荷を最小化するよう選定されている
- 時間の経過とともに、ネットワークの特性は変化していくことが想定される
 - そのため、実装が採用する値も将来的に変化していくことが想定される
- [MAY] 実装者は、上記の推奨値とは異なる値を使用してもよい
  - ただし最小遅延値など、本仕様の規範的要件に違反しないこと

## 10. Limitations
- Happy Eyeballsは、トランスポート層 (TCPやQUIC) における初期接続失敗に対応できる
  - それ以外の障害や性能上の問題は、選択された接続に依然として影響を与える可能性がある

### 10.1. Path Maximum Transmission Unit Discovery
- TCPのみを使用する接続 (TLS他の上位プロトコルを使用しない場合) では、接続の競争はTCPハンドシェイク中にのみ行われる
  - そのような接続において、小さなパケットによるハンドシェイクは成功しても、
    ネットワーク上で小さなMTUが設定されている場合は
    大きなパケットが破棄されたり不適切に処理されたりする問題が発生する可能性がある
    - この種の問題は、ネットワーク上の IPv6 通信に限定して発生する場合もある
  - この問題はTLS利用時にも発生し得るが、多くのTLSハンドシェイクで使われる比較的大きなメッセージにより、
    最大サイズのTCP セグメントが配送可能であることを確認できる場合が多い
  - QUIC接続では少なくとも1200バイトの最小MTUが保証されている
    - それより大きな値が利用できない可能性はある
  - これらの問題の解決は本書の対象範囲外であり、提案される手法の一つとして
    Packetization Layer Path MTU Discoveryの利用がある

### 10.2. Application Layer
- DNSが複数のアプリケーションサーバに対応する複数アドレスを返した場合でも、
  アプリケーション自体がそのすべてのサーバ上で正常に稼働し、機能しているとは限らない
  - 一般的な例としてTLSやHTTPがある

### 10.3. Hiding Operational Issues
- Happy Eyeballsは実運用においてネットワーク上の問題を見えにくくしてしまうことがある
  - e.g. 設定ミスによって特定ネットワーク上でIPv6が継続的に失敗している一方IPv4 は正常に動作している場合、
    Happy Eyeballsによってその問題に気付きにくくなる可能性がある
- ネットワーク運用者は、すべてのアドレスファミリの機能性を確認するため、外部的な監視手段を導入することが推奨される

## 11. Security Considerations
- アプリケーションは、安定したホスト名からアドレスへの対応関係に依存し、
  何らかのセキュリティ特性を保証しようとするべきではない
  - DNSの結果は問い合わせのたびに変化する可能性があるため
  - Happy Eyeballsによって、同一ホスト名への後続接続が異なるIPアドレスを使用する可能性はより高くなる
- HTTPを利用する場合、HTTPSリソースレコードはクライアントがオリジンへ接続する際にHTTPSを必須とするべきことを示す
  - RFC 9460
  - 能動的攻撃者はHTTPSリソースレコードの正常な配信を妨害することで、ダウングレード攻撃を試みる可能性がある
- クライアントが安全でないDNSメカニズムを使用している場合、通信経路上の攻撃者がHTTPSリソースレコードを破棄しうる
  - クライアントはそれが攻撃によるものなのか、HTTPS問い合わせに応答しないリゾルバによるものなのかを区別できない
  - [MUST NOT] RFC 9460に記載されているような暗号学的に保護されたDNSメカニズムを使用する場合、
    SVCB-reliantクライアントおよびSVCB-optionalクライアントのいずれも有効なHTTPS 応答を受信していない限り、
    TCPハンドシェイク完了後に暗号化されていないデータを送信してはならない
  - HTTPS応答が否定的応答でない場合、これらのクライアントは先へ進む前にTLSハンドシェイクを完了する必要がある
